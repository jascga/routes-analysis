## 🎯 项目启动阶段（2026-06-23）
### 需求确认
1. 项目目标：新建`routes-analysis`项目，实现华为交换机路由表多场景分析，复用`routescompare`项目的路由解析模块
2. 首个场景需求：
   - 输入：`display interface description` + `display ip routing-table protocol bgp`
   - 功能：把路由表中的接口替换为对端设备名，设备名格式为`区域-类型-序号-管理IP`，管理IP前字符串一致视为同组平行设备
   - 输出：分析负载分担到多组平行设备的路由，生成Excel报告
3. 复用方式：直接拷贝routescompare的parser.py和models.py模块到新项目，不做依赖集成
4. 输出形态：CLI + Excel报告
### 分组规则确认
1. 设备名按最后一个`-`拆分，去掉IP段作为分组键
2. 特殊规则：若第二段以`nc`开头且包含`_cnt`，保留`_cnt`，去掉后缀序号，同nc不同cnt视为同组
3. 不规范设备名（不含`-`）单独分组，标黄警告
4. 分组时忽略第三段（设备类型），同区域同序号不同设备类型视为同组

## 🧩 功能开发阶段（2026-06-23 ~ 2026-06-24）
### 场景1（多组分担分析）开发
1. 项目结构：`analyzer.py`负责分组分析逻辑，`exporter.py`负责4Sheet Excel导出（汇总/命中明细/所有路由/平行设备组清单）
2. CLI设计：子命令`multi-group`，支持多文件批量分析、自定义最小命中组数、编码指定、`--no-excel`只输出终端摘要
3. 测试：27个单元测试全过，端到端验证符合预期
### 场景2（多设备路由比较）整合
1. 整合策略：原`routescompare`项目废弃，核心代码（`comparator.py`/`excel.py`）迁移到`routesanalysis/comparison/`子目录
2. 子命令设计：新增`compare`子命令，参数完全兼容原routescompare，用户零迁移成本
3. 测试：迁移37个单元测试+11个测试用例，总用例数64个，全过

1. 问题：PyInstaller onefile模式下相对导入失败
2. 解决方案：把所有模块导入从相对导入改为绝对导入，`main.py`作为入口

## 🚀 未来优化讨论（2026-06-25）
### 优先级排序
1. **第一优先级（解决业务痛点）**：
   - 输入层抽象：支持多厂商/多数据源（SSH/网管/压缩包）
   - 规则配置化：YAML配置分组规则、差异阈值、忽略规则，不用改代码
2. **第二优先级（提升扩展性）**：
   - 分析层插件化：新增场景不用改核心代码，自动生成子命令
   - 统一输出层：支持多格式（CSV/HTML/JSON）、多目的地（邮件/企业微信）
   - 缓存+并行分析：大文件/批量场景速度提升10倍+
3. **第三优先级（体验增强）**：
   - 数据模型抽象：支持全协议路由分析
   - 轻量Web UI：在线分析、下载报告
   - 内置规则引擎：自定义告警规则，自动生成检查报告
### 现有解析库调研
- 优先选型：`TextFSM + ntc-templates`，轻量、华为命令模板全、打包体积小
- 备选：`huawei-parser`（华为专用）、`Scrapli Community`（SSH+解析一站式）
